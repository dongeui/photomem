## Problem Statement

Personal photo archives on local servers/NAS (100GB–TBs) have no meaningful semantic search.
Existing self-hosted tools (Synology Photos, Immich) offer CLIP-based search but it's English-only and tag-oriented.
Cloud solutions (Google Photos, Apple Photos) have excellent NL search but require surrendering privacy.

The specific pain: a Korean user types "4년 전 홍콩 여행 사진" (Hong Kong trip photos from 4 years ago)
into their local photo tool — and gets nothing. The same query in Google Photos works perfectly.
This product closes that gap, entirely offline.

## Demand Evidence

- User personally experiences this pain daily using Synology Photos on a NAS with TBs of photos
- Synology Photos has facial recognition but zero semantic/NL search capability
- Workaround is manual folder browsing — regularly fails, photos effectively lost
- The multilingual gap is real and unserved: Immich's CLIP search is English-trained;
  Korean/Japanese/Chinese queries return poor results across all existing self-hosted tools

## Status Quo

Current workflow:
1. Open Synology Photos
2. Try to recall approximate date → browse date folders
3. If date unknown: give up OR manually scroll through hundreds of photos
4. Cost: 10-30 minutes of frustration per search, frequently ending in failure

Existing self-hosted alternatives evaluated:
- Immich: CLIP search works for simple English queries, but no temporal+location+language combos
- PhotoPrism: AI classification (tags), no conversational/NL search
- Synology Photos: Face recognition only, no semantic search

## Target User & Narrowest Wedge

**v1 target user:** Privacy-conscious tech user in Korea/Japan/China who:
- Runs a NAS or home server (Synology, QNAP, or custom Linux box)
- Has 100GB–2TB of photos accumulated over 10+ years
- Refuses to upload photos to Google/Apple for privacy reasons
- Speaks Korean, Japanese, or Chinese as primary language
- Comfortable with CLI install or Docker

**Wedge:** A photo search engine installable in one command that handles
multilingual NL queries ("4년 전 홍콩 여행") on an existing local photo library.
Specifically: Korean/English/Japanese/Chinese natural language → accurate photo results.

**v2+ path:** Desktop app (Mac menu bar) targeting non-technical family users,
once the search quality thesis is proven.

## Constraints

- Must run 100% offline — zero network calls during indexing or search
- Must work on existing photo libraries (no migration, no photo duplication)
- Must support NAS sources (SMB/NFS mount or Synology API) AND local drives
- CLI install (brew/pip) for Mac; Docker for Windows
- Web UI runs on localhost — any browser, any OS
- Korean, English, Japanese, Chinese as first-class search languages

## Premises

1. **Privacy is the entry criterion** — self-hosted is non-negotiable for this user.
   Privacy gets them in the door; search quality keeps them.
2. **v1 is for tech-savvy users** (CLI/Docker), with a path toward non-technical users later.
   Non-technical family users are a v2+ opportunity, not the initial target.
3. **Architecture: local indexing + background AI pipeline + localhost web UI** is correct.
   Photos are never moved. Analysis runs in the background. The web app is the product.
4. **The multilingual gap is the real moat.** Immich ships CLIP features fast.
   Korean/Japanese/Chinese NL search is the differentiator they cannot easily replicate
   (requires different training data and model selection).

## Approaches Considered

### Approach A: Search-First MVP
One-command install. Indexes a folder with CLIP + EXIF. Localhost search box + thumbnail grid.
No gallery, no albums, no management. Ships in days. Proves the core thesis.
Effort: S (human: ~2 weeks / CC+gstack: ~2 hours)
Reuses: open_clip, sqlite-vec, FastAPI, HTMX

### Approach B: Full Private Photo Service (chosen)
Complete standalone photo app: gallery, albums, map view, face grouping, NL search.
Multilingual support (KO/EN/JA/ZH) as primary differentiator.
Docker + localhost web app. Replaces Synology Photos for tech users.
Effort: XL (human: ~6 months / CC+gstack: ~3 weeks)
Reuses: Docker, Postgres + pgvector OR sqlite-vec, React/Next.js or SvelteKit

### Approach C: LLM-Native Photo Memory
Every photo gets a rich narrative description from a local vision LLM (LLaVA/Moondream).
Search uses a local LLM for context-aware queries ("show me more from that trip").
Differentiates significantly from CLIP. Requires 8GB+ VRAM or slow CPU inference.
Effort: L (human: ~2 months / CC+gstack: ~1 week)
Note: C's local LLM descriptions should be incorporated as a Layer 2 enhancement in B.

## Recommended Approach

**Approach B, built in phases. C's LLM descriptions as a Layer 2 enhancement.**

**Phase 1 (MVP — prove the thesis):**
- Language: **Python** (pip install + brew tap with pre-built binary via PyInstaller or Nuitka)
- Multilingual search: **Option 2 — query translation pipeline** (NOT multilingual CLIP at this stage)
  - User types Korean/Japanese/Chinese query
  - llama-3.2-1b via ollama translates to English (runs on Mac/Linux; on NAS, skip translation
    and require English queries for v1 if RAM < 8GB — documented limitation, not a blocker)
  - Standard CLIP (ViT-B/32 via open_clip) generates embedding for translated query
  - Vector similarity search via sqlite-vec
  - EXIF parsing: date + GPS → offline reverse geocode to city/country
    (using photon-geocoder local DB or reverse_geocoder Python package, ~200MB)
- SQLite schema (per photo row):
  - id, file_path, file_hash, created_at (EXIF), latitude, longitude, city, country
  - clip_embedding (blob, 512-dim float32)
  - indexed_at, status (pending/indexed/error)
- File sync: **inotify (Linux) / FSEvents (Mac) / polling every 5 min (NAS/Docker)**
  - On startup: scan configured paths, queue unindexed files
  - On new file detected: enqueue to background worker
  - Background worker: single-threaded CLIP inference loop (batched, 8 images at a time)
- Localhost web UI: search box, photo grid, filters: date range, location (city/country)
  - Remove "person" filter from Phase 1 — no face data until Phase 2
- Distribution: **Docker is the primary v1 path** (avoids PyInstaller/Nuitka packaging
  complexity with torch + sqlite-vec native extensions). Single `docker run` command.
  brew tap is v1.1+ once distribution is stable.
- Model download: runs once on first launch, downloads to ~/.photomem/models/ (~1GB total:
  CLIP ViT-B/32 ~350MB + llama-3.2-1b GGUF ~650MB). Progress bar in terminal. No auto-download
  without explicit consent on first run.
- Ollama failover: if ollama is not running or query translation times out (>3s), fall back
  to raw query (treat Korean/Japanese/Chinese text as literal keyword search). Warn user in UI.
- LAN access: localhost-only binding by default (127.0.0.1). Optional: user can set
  PHOTOMEM_HOST=0.0.0.0 to expose on LAN for NAS use — documented as "you are responsible for
  LAN security." No auth in Phase 1; Phase 2 adds optional password protection.
- macOS full disk access: onboarding wizard prompts user to grant Full Disk Access in System
  Settings → Privacy before first scan. App checks permission on startup and shows clear error
  if not granted (FSEvents will silently fail without this).

**Phase 1 error handling:**
- Missing EXIF date: use file modification time as fallback
- Missing GPS: city/country fields null; location filter excludes these photos
- Model load failure: fall back to keyword-only search (filename + folder name)
- Indexing backlog > 10,000 photos: show progress indicator in UI, search available immediately
  for already-indexed photos

**Revised indexing performance estimate:**
NAS ARM CPU (e.g., Synology DS923+ Ryzen R1600, ~2 img/sec with CLIP ViT-B/32):
- 10,000 photos ≈ 83 minutes (not 30). This is acceptable background processing.
- 100,000 photos ≈ 14 hours. Needs user expectation setting in onboarding.
- Mac with Apple Silicon (M1/M2): ~20-50 img/sec → 10,000 photos in ~3-8 minutes.

**Phase 2 (full gallery):**
- Priority order within Phase 2 (do not bundle into one sprint):
  1. Face detection + grouping (InsightFace or dlib) → enables person filter
  2. NAS support: SMB/NFS mount + Synology Photos API (required to reach NAS users)
  3. Background daemon: incremental scan, not full re-index; 30-min polling + file events
  4. Timeline view (by year/month) — first gallery feature after search is proven
  5. Map view + albums — lower priority, only after timeline validates Phase 2 direction
- Multilingual search upgrade to Option 1 (multilingual CLIP): trigger = if Korean/Japanese/
  Chinese P@10 accuracy is >20% lower than English P@10 on the benchmark query set.
  If parity is achieved with query translation, Option 1 upgrade deferred indefinitely.

**Phase 3 (LLM-native layer — future exploration, not committed):**
- LLaVA/Moondream for narrative photo descriptions (optional, requires GPU or patience)
- Conversational search: "show me more from that trip" with context window
- Memory layer: the app "knows" your photo history, not just indexes it
- No validation criteria defined yet — Phase 3 is explored only after Phase 2 ships.

**Key technical decision — multilingual search:**
Option 1: Multilingual CLIP (mCLIP, or OpenCLIP multilingual checkpoints)
  → Query in Korean → embedding in Korean → compare against image embeddings
  → Requires multilingual-capable vision-language model at indexing time

Option 2: Query translation pipeline
  → User types in Korean → local LLM translates to English → CLIP search
  → Simpler, but adds latency and a local LLM dependency

Recommendation: **Phase 1 uses Option 2 (query translation).** This is a Phase 1 decision,
not a blocker. Multilingual CLIP model selection (Option 1) is a Phase 2 concern,
after the search quality thesis is validated with real user queries.

**Phase 1 multilingual CLIP dependency is NOT a blocker.** The dependency listed
in the Dependencies section refers to Phase 2 model selection, not Phase 1.

## Open Questions

1. Which multilingual CLIP checkpoint has the best Korean/Japanese/Chinese support?
   (ALIGN, multilingual-CLIP, or BLIP-2 with multilingual encoder)
2. What's the indexing speed on 1TB of photos on typical NAS hardware? (NAS CPUs are slow)
3. How do users without GPU handle LLaVA descriptions in Phase 3? CPU-only inference
   is feasible but slow — need a "light mode" for GPU-less users.
4. Synology Photos API vs. direct NAS mount — which is more reliable for keeping
   the index in sync with new photos?

## Success Criteria

- "4년 전 홍콩 여행 사진" returns the right photos (top 10) within 2 seconds
- Korean/Japanese/Chinese queries perform within 20% of equivalent English queries
- One-command install works: `brew install photomem` or `docker compose up`
- Indexing 10,000 photos completes within 90 minutes on NAS ARM CPU (2 img/sec baseline);

- Zero network calls after initial model download (fully offline operation)

## The Assignment

Before writing a single line of code: **set up Immich with 1,000 of your actual photos,
try 20 real searches you'd actually want to do (in Korean and English), and score how
many return the right results in the top 10.**

Then try the same 20 searches using a quick Python script with multilingual sentence
transformers on the same 1,000 photos.

That A/B comparison is your thesis test. If multilingual sentence transformers beat
Immich by more than 30%, you have a product. If they don't, you need to rethink
the model choice before building the UI.

### Architecture (Phase 1 — confirmed)

```
Docker Container
├── FastAPI + uvicorn (single process)
│   ├── GET /            → full page (Jinja2)
│   ├── GET /search?q=   → HTMX fragment (text CLIP → sqlite-vec → grid HTML)
│   ├── GET /thumb/:id   → serve from cache volume
│   └── GET /status      → HTMX fragment (indexing progress)
├── asyncio background task
│   └── ProcessPoolExecutor (max_workers=1 default)
│       └── watchdog event → Pillow thumbnail → piexif EXIF
│           → reverse_geocoder → visual ONNX CLIP → SQLite WAL write
├── SQLite (WAL mode)
│   ├── photos: id, file_path, file_hash, created_at, lat, lng, city, country,
│   │          clip_embedding (512-dim float32), indexed_at, status, error_msg
│   └── index_queue: pending jobs
└── Cache Volume
    ├── thumbnails/   (JPEG 300px, ~50KB each)
    └── models/       (text_encoder.onnx ~50MB, visual_encoder.onnx ~350MB)
Volumes:
  /your/photos:/data/photos:ro
  photomem_cache:/app/cache:rw
```

**Key clarification from D1:** LLM/ollama is NOT Phase 1. Phase 1 is pure CLIP pipeline.
Phase 1 multilingual search works via CLIP's native text embedding capability (no translation needed).

### Must-Fix Before First Run (Sprint 1)

1. **ProcessPoolExecutor worker init** — load visual ONNX in `initializer=_worker_init`, not in main process. Model is not picklable; passing it as argument silently fails.
2. **WAL autocheckpoint** — `PRAGMA wal_autocheckpoint=500` at DB init. Without this, WAL grows unboundedly during 100K photo initial index.
3. **Indexer resume on restart** — on startup, re-queue all `status='pending'` rows. Without this, container restart loses all partially-indexed progress.
4. **Mount validation** — check `/data/photos` non-empty at startup; print clear error + exit(1) if not. Silent empty library is the #1 user confusion.
5. **ONNX model checksum** — SHA256 verify after download. Corrupted model produces silently wrong embeddings.

### Critical Failure Modes

- **Silent worker crash**: ProcessPoolExecutor dies but indexer reports "running". Add heartbeat.
- **Empty search results state**: Zero results → empty `<div>`, user thinks search is broken. Add message.
- **PHOTOMEM_MAX_WORKERS**: Default=1 for NAS (2 workers = ~1GB model RAM). Document env var.

### Performance (confirmed meets SLA)

- Search at 100K photos: ~75ms total (text encode + KNN + HTML) ✅ (SLA: 2s)
- Indexing 10K photos on NAS ARM: ~55-83 min ✅ (SLA: 90 min)
- Docker image size: ~500MB (ONNX vs PyTorch 2GB) ✅
- Memory: ~680MB single worker ✅ (NAS 2-4GB RAM)

```
app/
  main.py         # routes only
  db.py           # SQLite init + CRUD
  indexer.py      # watchdog + pool + CLIP pipeline

    _status.html   # HTMX fragment
```
